## 一、Learn 项目架构升级 —— v2 lazy loading

### 起因
原来 `learn_data/sql/course.js` 是 **2246 行单文件**，Python 是 **3278 行**。改一节课要在几千行里翻找，可维护性几乎为零。

### 方案选择
讨论了 4 个 Plan（A 切章节 / B 一课一文件 / C 浏览器构建 / D 组合），最终选 **Plan C 的浏览器版本**：因为零后端约束，不能用 node 构建。所有课程数据通过 `<script>` 标签按需加载。

### 最终架构

```
loadCourse(slug)         ← 进课程时只加载课元数据 + lesson 索引（~几 KB）
   ↓
loadLesson(slug, id)     ← 进单节课时才加载该课内容
   ↓
loadSchema(slug, name)   ← lesson 引用了 schema 时才加载
```

### 关键设计：`@@key` 文本格式

每节课是一个 JS 文件，调用 `LEARN.lesson(slug, id, raw)`。raw 是模板字符串，里面用自创的 `@@key` 语法：

```js
LEARN.lesson('sql', 1, `
@@schema c1_schema
@@expectedSql SELECT name FROM students
@@checkOrder false

@@title:zh Part 1 · SELECT —— 选哪些列
@@title:en Part 1 · SELECT — pick the columns

@@intro:zh
<p class="lead"><strong>SELECT</strong> 是 SQL 的第一关键字...</p>
`);
```

- **`@@`（双 at）**做字段标记，避开 `@property` / `@app.route` 这种 Python 装饰器在代码示例里出现的冲突
- **单行字段** `@@key value`，**多行字段** `@@key\n<内容>`
- **双语** 用 `@@key:zh` / `@@key:en` 后缀
- **多行块尾随 `\n`** 通过"在下个 `@@key` 前留空行"编码（用于 starter 这类需要尾换行的字段）

### Schema 共享机制

SQL 课原本 42 节课各自 inline 写 `setup` SQL，导致 `students` 表被 CREATE 了 10 次，`books` 6 次。新架构把 schema 提取到 `learn_data/sql/schemas/<name>.js`，lesson 用 `@@schema c1_schema` 引用。**保守去重原则**：只有完全字节相同的才合并；命名 `c<最小lesson_id>_schema`，明显共享的（如 final challenge 系列共用的 `final_schema`）给语义化名字。

最终结果：
- SQL：42 个 inline → 29 个独立 schema 文件（5 组共享）
- Python：每课自包含，无 schema

## 三、C 语言课程 —— 最大的坑

### 候选方案排查（依次试错）

| 方案 | 结果 |
|---|---|
| **JSCPP** | malloc / struct / 函数指针**全坏**，只能跑最基础 ❌ |
| **TCC.wasm**（eliot-akira） | 是**交叉编译器**（cross），编译 i386 但浏览器不能执行 i386 → `-run` 直接报错 ❌ |
| **emception** | clang in WASM，**真编译器**，但是 webapp 不是 library ⚠ |

### 最终方案：emception + 自定义 adapter

**架构**：
```
learn.html (parent)
   │ 创建隐藏 iframe → lib/runtime/webC/iframe.html (同源)
   ▼
iframe.html
   ├ 加载 main.bundle.js（upstream emception webpack 产物）
   └ 加载 postmsg-bridge.js（我们写的 adapter）
       ├ 等 window.emception（upstream demo 自动暴露）
       ├ 隐藏 upstream 自带 IDE UI
       └ 监听 postMessage {type:'run', code, stdin}
           └ 写 /working/main.c → emcc → 读 main.js → new Function 执行
```

**关键决策：mirror 整个 emception 到本地** (`lib/runtime/webC/`，~450 MB / 522 文件)
- 跨源 Worker 浏览器**硬性禁止**（即使 jsDelivr 有 CORS 头也不让），所以不能从 CDN 加载 main.bundle.js
- 同源镜像后所有 fetch 都同源，零跨源问题
- Cloudflare Pages 25 GB / 20k 文件配额扛得住

### 配套的 UX

1. **C-family modal warning**：manifest 里 C 课加 `family: 'c'` 标签。`route()` 检测到就先弹 modal："⚠ 你正在尝试访问 C 语言相关的资源" + 两个按钮（取消 / 加载资源）。点确认后 localStorage 持久化，下次不再弹
2. **下载进度条**：`postmsg-bridge.js` 用 `fetch + ReadableStream` 流式下载 main.bundle.js，实时报字节进度。M3 风格的 determinate / indeterminate 进度条
3. **左侧 Clang 加载卡**：加载完后**渐变成绿色 "✓ Clang 就绪"**（不消失），透出加载已完成的视觉信号
4. **Run 按钮三态机**：默认（▶ 运行）→ 编译中（灰色禁用）→ 完成（绿色 1.2 秒闪一下）→ 默认

### 失败的尝试：交互式 scanf

希望像 Python 那样 scanf 弹光标等用户输入。试了 `-sASYNCIFY=1` 让 wasm 可以 suspend：

- 第一坑：`fd_read` 没在 `ASYNCIFY_IMPORTS` 里 → 加上修了
- 第二坑：EXIT_RUNTIME 阶段 `proc_exit` 调用触发 ASYNCIFY 状态机崩溃，加 `proc_exit` 到 IMPORTS 反而让 scanf 第一次调用就崩（noreturn 函数被插桩破坏 wasm layout）
- 第三坑：尝试 `INVOKE_RUN=0 + callMain` 替代 EXIT_RUNTIME，scanf 入口阶段崩，原因不明
- 第四坑：cosmetic abort 过滤当成功，但用户根本没收到输入，只是被静默"完成"

**结论**：emception 的特定 emscripten build 上 ASYNCIFY 太脆弱，**放弃交互式 scanf**，回到"预填 stdin 模型"。stub 代码留着（`onInputRequest` / `termRequestInput` / `input-request/response` postMessage 协议），等浏览器普及 JSPI（Chrome 133+ 默认）后能用更稳的机制重做。

### 修了的 stdout 乱序问题

`printf("Hello"); scanf(...);` 提示语会被 libc 行缓冲卡住。两种解决：
1. 在 prompt printf 末尾加 `\n`（line buffer 自动 flush）
2. 调 `fflush(stdout);`

默认 demo 用了第一种。

### 修了的 stderr 错误信息连成一行

iframe 转发 emcc stderr 时没加换行符 → 多行编译错误挤一坨。修复：`withTrailingNl()` 包装。同时剥 ANSI 颜色码（`\x1b[32m` 这种）+ 过滤 `^[a-z_]+:(INFO|DEBUG):` 这种 emscripten 内部 noise（如 "shared:INFO: (Emscripten: Running sanity checks)"）。

### 默认 demo 程序

```c
#include <stdio.h>

int main(void) {
    char name[64];
    int  age;
    char *p = name;          // 指向 name 数组首元素的指针

    printf("What's your name?\n");
    scanf("%63s", p);        // 通过指针 p 把名字写进 name

    printf("How old are you?\n");
    scanf("%d", &age);       // &age 取 age 的地址 —— 也是个指针

    printf("Hello, %s! You are %d years old.\n", p, age);
    return 0;
}
```

stdin 预填 `Louie\n19`。一次点 Run 就能看到完整流程，覆盖：指针声明 / 通过指针写入 / 取地址符 / 数组与指针关系 / scanf / printf 多变量 / 行缓冲与换行。

## 四、Cloudflare 部署

### 踩的坑
我一直说"Cloudflare Pages 25 GB 没问题" —— 但 **Cloudflare 已经把新建的 Pages 项目偷偷迁移到 Workers Static Assets 体系**了。Dashboard 字段不一样了：

| Pages（老） | Workers Static Assets（新） |
|---|---|
| Build command + Output directory | Build command + **Deploy command** + **Version command** |
| 无 `wrangler.toml` 也能部署 | **必须**有 `wrangler.toml` 否则 deploy 不知道部署啥 |

### 最终修复
仓库根目录加两个文件：

**`wrangler.toml`**：
```toml
name = "homepage"
compatibility_date = "2025-05-08"

[assets]
directory = "./"
```

**`.assetsignore`**（排除 `.git/`、笔记 md 等不该上线的）：
```
.git/
.gitignore
wrangler.toml
.assetsignore
README.md
LICENSE
**/learn_readme.md
node_modules/
.DS_Store
...
```

### Cloudflare Workers Static Assets 实际配额
- **20,000 文件**（我们 530）
- **25 MiB 单文件**（我们最大 23 MB）
- **~9.5 GB 总体积**（我们 470 MB）

全部远低于上限。

## 六、几个值得记住的教训

1. **零后端约束很硬**：每次想用 node 构建脚本都要忍住，回到"浏览器内能跑"的设计
2. **跨源限制比想的严**：CORS 解锁了 fetch / image / xhr，但**解锁不了 Worker** —— 这点害我兜了半天，最后只能 mirror
3. **emscripten ASYNCIFY 脆**：不是参数能调通的，state machine 一进入就基本无法预测。教训：将来类似的"在 wasm 里跑外部异步交互"，优先考虑 JSPI 而非 ASYNCIFY
4. **Cloudflare 在悄悄迁移**：Pages → Workers Static Assets。新项目默认用新体系，老知识可能过时
5. **Mirror 永远是兜底解**：复杂跨源 / 配额 / CDN 不可控时，就把外部依赖打包进自己仓库。450 MB 听着大，但 Cloudflare CDN 全球缓存零成本
